<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Scheduled jobs → health table | Watchdog reads health rows; a missing row is the signal of a dead component crossing into the monitoring layer |
| DB tables → DR snapshot bucket | Table contents cross into an encrypted backup; empty critical tables would produce a corrupt-but-committed backup |
| Cron + ingest triggers → Telegram | Two triggers can race to send the same morning brief |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-07 | Tampering / DoS | DR snapshot of empty critical tables | mitigate | L3 `assert_snapshot_sane()` raises EmptySnapshotError and aborts before commit, preventing a corrupt backup overwrite (RESEARCH.md §Security Domain "Corrupt backup commit"); this plan asserts the guard is present and called |
| T-03-08 | Repudiation | a silently-dead component with no health row | mitigate | L4 `compute_alerts()` iterates expected components (THRESHOLDS_HOURS) so a missing row alerts, not just stale rows; asserted present + unit-tested |
| T-03-09 | Denial of Service | duplicate morning brief from double trigger | mitigate | PIPELINE-04 `_brief_already_sent_today()` DB check skips a second send within the window; this plan adds an explicit unit test for it |
| T-03-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this plan (RESEARCH.md §Package Legitimacy Audit: none); no install task, no slopcheck gate needed |
</threat_model>
