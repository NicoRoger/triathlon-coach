## 4. Fail point documentati ma NON modificati (per scelta / rischio)

### 4.1 — MCP server: auth bypass 🔴 HIGH (documentato, non toccato)
`workers/mcp-server/src/index.ts:358`:
```ts
const isOAuthRequest = !auth; // Claude.ai dopo OAuth non manda bearer
if (!isBearerValid && !isOAuthRequest) { /* 401 */ }
```
Una request **senza** header `Authorization` viene trattata come autorizzata
(OAuth) **senza alcuna validazione del token OAuth**. Chiunque conosca l'URL del
worker può quindi invocare ogni tool: lettura di tutti i dati (`get_recent_metrics`,
`get_dashboard_data`, …), scrittura piano (`commit_plan_change`,
`commit_mesocycle`) e `force_garmin_sync`.

**Perché non modificato ora:** chiudere il buco richiede validare un vero OAuth
access token (o richiedere sempre il bearer), il che può rompere il connettore
Claude.ai funzionante. Scelta dell'atleta: documentare e affrontarlo a parte.

**Mitigazione proposta** (futura):
- Validare l'OAuth access token contro l'authorization server, **oppure**
- Richiedere sempre `Bearer ${MCP_BEARER_TOKEN}` e configurare Claude.ai per inviarlo, **oppure**
- Almeno, finché non risolto: trattare l'URL del worker come segreto e ruotarlo se sospetti esposizione.

### 4.2 — Strava sync disattivato 📋 (documentato, non riattivato)
`.github/workflows/ingest.yml:57-59`: lo step Strava è interamente commentato →
Garmin è single point of failure per l'ingest, senza backup. Scelta dell'atleta:
lasciarlo disattivato (probabile motivazione costi/rate-limit). Il modulo
`coach/ingest/strava.py` resta comunque funzionante (e ora con guard anti-loop, A6).

### 4.3 — `session_analyses.activity_id` TEXT, non FK 📋
Memorizza l'`external_id` (stringa) anziché un FK verso `activities`. Cambiarlo in
FK rischia di rompere il matching dell'MCP `get_session_review_context` (che cerca
prima per `external_id`). È una scelta di design: documentato, non modificato.

### 4.4 — Timezone dei cron 📋 (basso impatto)
Diversi workflow (`morning-briefing`, `weekly-review`, `debrief-reminder`,
`proactive-check-in`) usano cron in UTC con commenti che assumono ora legale
(Europe/Rome = UTC+2). In inverno (UTC+1) gli orari slittano di 1h. Impatto basso
(notifiche, non logica). Da rivalutare semestralmente o documentare come noto.

### 4.5 — Raccomandazione "GRANT to anon" scartata
Un'analisi automatica aveva suggerito `GRANT SELECT,INSERT,UPDATE ... TO anon` su
tutte le tabelle come risposta all'email Supabase. **Scartata**: indebolirebbe la
sicurezza esponendo i dati alla Data API pubblica. Per un'app service_role-only la
risposta corretta è abilitare RLS (vedi §2).
